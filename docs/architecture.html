Architecture

How Design Storm is built

Design Storm is a full-stack Rust web application for AI-directed design exploration. A monolithic Axum server renders HTML templates, streams real-time updates over SSE via Datastar, and orchestrates agentic AI generation through the lash runtime. The frontend is vanilla TypeScript with a WebGL-powered infinite canvas.

Philosophy

Server is the source of truth. HTML-over-the-wire via Datastar SSE. Minimal client state. The backend drives the UI by patching elements and signals.

Stack

Rust 1.93 · Axum 0.8 · sqlx + PostgreSQL · Askama templates · Datastar RC.8 · TypeScript · Bun · WebGL2 · Fly.io

01 / System Overview

High-Level Architecture

A single Rust binary serves the entire application — static assets, server-rendered HTML, REST APIs, and SSE streams. The AI generation pipeline runs in-process via the lash agent runtime, writing artifacts to an isolated filesystem workspace.

02 / Backend

Rust + Axum Server

A single-binary Axum application handles routing, template rendering, database access, authentication, and AI orchestration. All state is shared via an Arc<AppState>.

Core Dependencies

Datastar SSE Pattern

Long-running operations (generation, node creation, deletion) return SSE streams. The Datastar client automatically patches the DOM and updates signals from these events. 

sequenceDiagram
    participant B as Browser
    participant D as Datastar Client
    participant S as Axum Server

    B->>D: Click button (data-on:click="@post")
    D->>S: POST /storms/generate
    S-->>D: event: datastar-patch-signals
data: {_generating: true}
    Note over S: Run lash agent...
    S-->>D: event: datastar-patch-elements
data: selector #roots-list
data: mode inner
data: elements <html>
    S-->>D: event: datastar-patch-elements
data: selector #storm-runs
data: mode outer
data: elements <html>
    S-->>D: event: datastar-patch-signals
data: {_generating: false, _latestRunId: "..."}
    D->>B: DOM morphed + signals updated

The server uses PatchElements and PatchSignals from the datastar Rust crate, streamed through Axum's Sse response type with async_stream::stream!.

03 / Frontend

TypeScript + Datastar + WebGL

A hybrid approach: server-rendered HTML with Datastar directives for reactivity, plus vanilla TypeScript for the infinite canvas, node interactions, and complex UI state.

Architecture
Server-Rendered Layer

Askama templates generate initial HTML. Datastar attributes (data-signals, data-show, data-on) provide reactivity without a framework.

Client State Layer

A single StormState object manages runs, board nodes, edges, positions, and pointer interactions. Hydrated from DOM data attributes on page load.

Canvas Layer

WebGL2 procedural background rendered as one seamless presentation surface, backed by cached offscreen chunks. DOM-based node rendering uses CSS transforms for positioning, and SVG renders the connection lines.

Board Interaction Model
stateDiagram-v2
    [*] --> Idle
    Idle --> Pan: Space + drag / Hand tool
    Idle --> DragNode: Pointer down on node
    Idle --> ResizeNode: Drag resize handle
    Idle --> Wire: Pointer down on edge handle
    Idle --> RadialMenu: Right-click / Backtick

    Pan --> Idle: Pointer up

Pointer state is modeled as a discriminated union with modes: pan, drag, drag-board-node, wire, and resize. The board now uses Figma-style navigation: wheel pans, Ctrl/Cmd + wheel zooms, and the bottom tool dock controls select vs hand mode. Radial-created nodes spawn at the invocation point, while toolbar-created nodes spawn centered in the current viewport.

Build Pipeline
Bun TypeScript No framework No bundler plugins
bun build ./frontend/app.ts --outfile ./static/app.js --target browser --minify

Single entry point, single output file. Bun handles TypeScript compilation and bundling. The only npm dependency is @clerk/clerk-js for client-side auth.

04 / Generation Pipeline

The Lash Agent Runtime

Design artifacts are generated by an agentic AI runtime called lash. The server creates an isolated filesystem workspace, configures the agent with restricted capabilities and custom tools, then runs iterative turns until the artifact is submitted.

Agent Configuration
Sandboxed Tools

workspace_list, workspace_read, workspace_write — confined to the run directory. render_result and submit_result for artifact lifecycle. Optional search_web and fetch_url via Tavily.

Disabled Capabilities

No shell access, no filesystem beyond workspace, no memory persistence, no task delegation, no skill loading. The agent can only modify its own workspace and call web search if configured.

Provider Abstraction

Supports Codex (OpenAI), Claude, Google OAuth, and a generic OpenAI-compatible endpoint. Provider credentials encrypted at rest with AES-256-GCM. Model tier selection per provider.

Prompt Engineering

Custom system prompt sections override identity, personality, execution contract, tool access, guidelines, and error recovery. The agent is instructed to produce static HTML design artifacts.

Node Graph Generation

The stormboard's generate node collects wired inputs and builds a composite prompt. When the user clicks Run, the backend resolves all incoming edges: 

flowchart LR
    E["Entropy Node
Wikipedia article"] --> G["Generate Node"]
    I["Input Node
User text + images"] --> G
    D["Design Node
Previous artifact"] --> G
    G --> |"Run"| R["New Design Node
Generated artifact"]
    G -.-> |"auto-wire"| R

Each source type contributes to the prompt differently: entropy provides reference URLs, user input provides direction text, and design nodes provide context from previous generations. Generate-node execution is server-owned and returns Datastar SSE patches that refresh the board HTML, roots navigator, and local generation signals.

Node Types
entropy

Random Wikipedia article. Content stores title and url. Can be re-rolled and locked.

user_input

Free-form text and pasted images. Content stores text and images (data URLs). Contenteditable.

Aggregation node. Accepts wired inputs from other nodes. Run button triggers AI generation. Output auto-wires to a new design node.

06 / Authentication

Clerk + Session Cookies

Auth Flow
sequenceDiagram
    participant U as User
    participant C as Clerk JS
    participant S as Axum Server
    participant K as Clerk JWKS

    U->>C: Sign in (hosted modal)
    C->>U: Clerk JWT token
    U->>S: POST /auth/session {token}
    S->>K: Validate JWT signature (JWKS)
    K-->>S: Valid
    S->>S: Upsert user record
    S->>S: Issue HS256 session JWT (7 day)
    S-->>U: Set-Cookie: designstorm_session
    Note over U,S: Subsequent requests use cookie
    U->>S: GET /app (cookie attached)
    S->>S: Verify session JWT
    S-->>U: Rendered page

Session cookies are HttpOnly, Secure, SameSite=Lax. The session JWT contains {sub: user_id, clerk_user_id, exp}. Every handler calls require_viewer() to authenticate.

Provider Credentials

LLM provider credentials are encrypted with AES-256-GCM before storage in PostgreSQL. The encryption key derives from the SESSION_SECRET environment variable. Supported flows:

Codex (OpenAI) — Device OAuth

Polling-based device code flow. User visits a URL, enters a code, and the server polls until authorization completes. Stores access token + optional account ID.

Fallback — Generic OpenAI

API key + base URL for any OpenAI-compatible endpoint. Configured via environment variables. Used when no user-specific provider is connected.

Runtime Configuration
Compute

Fly.io shared CPU, 1 vCPU, 1 GB RAM. Auto-scales to zero when idle. Health checks via GET /healthz every 15 seconds.

Storage

Workspace artifacts are written locally, zipped, and uploaded to S3-compatible object storage for durable preview recovery. Database migrations run automatically on startup via sqlx.

Networking

Force HTTPS. Single port 8080. Axum serves static files, docs, templates, and API routes from the same listener. SSE with 15-second keepalive.

Observability

Structured logging via tracing with span-level HTTP request traces (tower-http TraceLayer). Client telemetry endpoint for frontend errors.
